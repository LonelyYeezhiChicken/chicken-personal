# 📊 管制界限與規格界限

本章節只做一件事：釐清**管制界限**與**規格界限**各自回答什麼問題，以及四種常見衝突場景怎麼判讀。入門定義見 [`terminology`](../terminology.md)。

## 讀完本篇你能回答

- UCL/LCL 與 USL/LSL 的來源與職責有何不同？
- 為什麼「製程受控」不等於「產品合格」？
- 能不能為了少報警而手動拉寬管制界限？

## 1. 管制界限：製程在說話

管制界限（$UCL$ / $CL$ / $LCL$）由**歷史數據的統計特徵**決定，不代表客戶需求。

| 項目 | 說明 |
|------|------|
| 常見設定 | $\mu \pm 3\sigma$ |
| 機率意義 | 常態下約 $99.73\%$ 點落在界內 |
| 回答的問題 | 製程還穩不穩？（OOC） |

$3\sigma$ 是統計與成本之間的折衷：太窄虛警多（型一錯誤），太寬漏警多（型二錯誤）。

## 2. 規格界限：客戶在說話

規格界限（$USL$ / $Target$ / $LSL$）由**工程設計或客戶需求**訂定，與歷史統計無關。

| 項目 | 說明 |
|------|------|
| 職責 | 判斷產品合格或廢品 |
| 回答的問題 | 產品合不合格？（OOS） |

## 3. 四種衝突場景

| 場景 | 現象 | 結論 | 建議動作 |
|------|------|------|----------|
| 理想 | $6\sigma$ 遠小於規格寬度 | 受控且高 $C_{pk}$ | 維持監控 |
| 能力不足 | 管制界超出規格界 | 受控仍會產生廢品 | 改善製程能力 |
| 預警 | 點出管制界但在規格內 | 產品合格、製程不穩 | 預防性介入 |
| 雙重違規 | 同時 OOC + OOS | 最高優先 | Hold / 停線評估 |

## 4. 界限必須自動計算

:::info 實務提醒
為了減少報警而**手動拉寬**管制界限，只會掩蓋不穩定，無法解決 OOS。界限應由統計引擎依歷史數據自動更新。
:::
